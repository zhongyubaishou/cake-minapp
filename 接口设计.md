# 蛋糕定制微信小程序 — 接口设计文档

## 一、接口规范

### 1.1 通用返回结构

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### 1.2 错误码约定

| code | 含义 |
|------|------|
| 0 | 成功 |
| 400 | 参数错误 |
| 401 | 未登录 / Token 无效 |
| 403 | 无权限 |
| 404 | 资源不存在 |
| 409 | 业务状态冲突（如订单状态不允许此操作） |
| 500 | 服务器内部错误 |

业务错误通过 `code` 区分，前端仅展示 `message`。

### 1.3 鉴权方式

| 端 | 方式 |
|---|---|
| 小程序端 | Header `Authorization: Bearer {userToken}` |
| 后台管理端 | Header `Authorization: Bearer {adminToken}` |

### 2.13 计算配送距离和配送费

```
POST /api/delivery/calculate
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| address | string | 是 | 配送地址（用于日志和校验） |
| longitude | number | 是 | 用户地址经度 |
| latitude | number | 是 | 用户地址纬度 |

**返回：**

```json
{
  "code": 0,
  "data": {
    "distanceKm": 1.243,
    "deliveryFee": 2.00,
    "isInRange": true,
    "message": ""
  }
}
```

**业务规则：**
- 距离用 Haversine 公式计算
- 存储 `distanceKm` 到 `DECIMAL(6,3)`（如 1.243）
- 配送费 = `ROUND(distanceKm) × 2`
- 四舍五入后为 0km 时，配送费为 0 元；如需最低 1km 起收，在门店设置中另行配置
- 范围判断用实际距离：`distanceKm > 5` 时 `isInRange = false`

### 2.14 创建订单

```
POST /api/orders
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| source | enum | 是 | PRODUCT / CUSTOM / CASE |
| productId | int | source=PRODUCT时必填 | 商品 ID |
| caseId | int | source=CASE时必填 | 案例 ID |
| categoryId | int | 是 | 分类 ID |
| buyPound | number | 是 | 购买磅数 |
| flavor | string | 是 | 口味 |
| theme | string | 是 | 主题风格 |
| blessingText | string | 否 | 祝福文字 |
| plateCount | int | 是 | 碟子数量 |
| candleCount | int | 是 | 蜡烛数量 |
| referenceImages | array | 否 | 参考图片 URL 数组 |
| pickupType | enum | 是 | SELF_PICKUP / DELIVERY |
| appointmentTime | datetime | 是 | 预约时间 |
| address | string | 配送必填 | 配送地址 |
| addressLng | number | 配送必填 | 地址经度 |
| addressLat | number | 配送必填 | 地址纬度 |
| receiverName | string | 否 | 收货人姓名（默认用户昵称） |
| receiverPhone | string | 配送必填 | 收货人手机号（默认用户手机号） |
| userRemark | string | 否 | 用户备注 |

**source 与关联 ID 校验表：**

| source | productId | caseId | 校验 |
|--------|-----------|--------|------|
| PRODUCT | 必填 | 必须为空 | "商品下单缺少 productId" |
| CASE | 必须为空 | 必填 | "案例定制缺少 caseId" |
| CUSTOM | 必须为空 | 必须为空 | - |

**前置校验（任一不通过返回 400）：**

| 校验项 | 错误信息 |
|--------|---------|
| `user.phone` 不为空 | "请先绑定手机号" |
| `appointmentTime >= now() + 6小时` | "预约时间需至少提前6小时" |
| 配送订单：`distanceKm <= 5` | "超出配送范围(5km)，请选择到店自提" |

**返回：**

```json
{
  "code": 0,
  "data": {
    "orderId": 1,
    "orderNo": "20260607000001",
    "status": "WAIT_ACCEPT"
  }
}
```

**内部处理：**
- 订单状态初始化为 `WAIT_ACCEPT`
- `is_outside_hours = (appointmentTime < store_config.business_hours_start 或 > store_config.business_hours_end) ? 1 : 0`
- 快照用户昵称、手机号、分类名、商品名/主图
- `actualMakePound = buyPound × 1.5`
- 配送订单写入 delivery_info
- 写入 operation_log

## 三、支付与退款回调（V1 暂不启用）

### 3.1 支付成功回调（V1 暂不启用）

```
POST /api/pay/wechat/notify
```

**V1 规则：** 当前版本不接入微信支付，不会收到微信支付成功回调。该接口可暂不注册路由，或注册后直接返回固定成功/禁用提示用于占位。

**订单状态更新来源：** 商家后台确认报价/接单接口负责将订单直接更新为 `PAID_WAIT_MAKE`，并写入 `pay_status = PAID`、`payment_method = OFFLINE`、`paid_at = accepted_at`。

### 3.2 退款回调（V1 暂不启用）

```
POST /api/refund/wechat/notify
```

**V1 规则：** 当前版本不调用微信原路退款，不会收到微信退款回调。退款、改期、取消等由商家线下协商处理。

**后续恢复微信支付时再启用：** 支付回调校验、金额核对、微信交易号写入、原路退款回调、退款失败处理等能力。


3. 生成 `out_refund_no`
4. V1 不调用微信支付退款 API；后续接入微信支付后恢复

7. 写入 operation_log

**注意：** 微信退款为异步，回调可能延迟 5-15 分钟。`REFUND_PROCESSING` 状态下前端应展示"退款处理中，请稍候"。

## V1 临时支付链路调整（最终口径）

因商家暂不注册个体工商户、暂不接入微信支付商户号，V1 当前支付链路调整如下。本节为最终口径，覆盖本文档中所有“待支付、微信支付、支付回调、微信原路退款”的旧描述。

### 1. 当前订单主流程

```text
用户提交订单 → 待接单
商家确认报价/接单 → 已支付/待制作
商家点击开始制作 → 制作中
制作完成：
  自提订单 → 待自提 → 已完成
  配送订单 → 待配送 → 配送中 → 已完成
```

### 2. 支付处理规则

1. V1 不接入微信支付，不调用 `uni.requestPayment`，不创建微信预支付单。
2. 商家在后台点击确认报价/接单后，系统直接将订单状态置为 `PAID_WAIT_MAKE`（页面文案：已支付/待制作）。
3. 该“已支付”是业务上的线下确认状态，不代表微信支付真实到账。
4. 后端可写入 `pay_status = PAID`、`payment_method = OFFLINE`、`paid_at = accepted_at`，用于驱动现有订单状态和报表逻辑。
5. 如需保留支付记录，可创建一条本地记录，状态建议为 `OFFLINE_CONFIRMED`；不得生成微信 `transaction_id`。
6. 报表中的销售额按“商家确认后的订单金额”统计，不等同于微信支付流水或真实到账金额。

### 3. 退款处理规则

1. V1 用户端不展示申请退款入口。
2. V1 后台不展示退款审核入口，退款列表可隐藏或置灰。
3. 退款接口和微信原路退款代码暂不启用；如保留接口，应返回“当前暂不支持线上退款”。
4. 线下退款、改期、取消等由商家与用户自行协商，不进入小程序自动退款流程。
5. 售后反馈仍可保留，但只作为问题反馈，不自动退款。

### 4. 后续恢复微信支付时的切换点

后续商家完成主体认证并开通微信支付后，再恢复以下链路：

```text
商家确认报价/接单 → 待支付
V1 跳过线上支付成功回调（V1 暂不启用） → 已支付/待制作
```

届时再启用微信预支付、支付回调、微信原路退款和真实支付流水统计。
