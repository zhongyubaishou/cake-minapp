# 蛋糕定制微信小程序 — 状态机设计文档

## 一、订单主状态机

### 1.1 状态流转图

```
                  ┌──────────────┐
                  │   未提交      │
                  └──────┬───────┘
                         │ 用户提交订单
                         ▼
                  ┌──────────────┐        ┌──────────┐
                  │ WAIT_ACCEPT  │───────▶│ CANCELED │
                  │   待接单      │        │  已取消   │
                  └──────┬───────┘        └──────────┘
                         │
                         │ 老板确认报价/接单
                         │ (根据取货方式选择目标状态)
                         │
              ┌──────────┴──────────┐
              ▼                     ▼
     ┌────────────────┐    ┌────────────────┐
     │  WAIT_PICKUP   │    │ WAIT_DELIVERY  │
     │    待自提       │    │   待配送       │
     └───────┬────────┘    └───────┬────────┘
             │                     │
             │ 确认取货            │ 开始配送
             ▼                     ▼
     ┌────────────────┐    ┌────────────────┐
     │   COMPLETED    │    │  DELIVERING    │
     │    已完成       │    │   配送中       │
     └────────────────┘    └───────┬────────┘
                                   │ 确认送达
                                   ▼
                          ┌────────────────┐
                          │   COMPLETED    │
                          │    已完成       │
                          └────────────────┘
```

### 1.2 状态定义

| 状态常量 | 中文文案 | 说明 |
|----------|----------|------|
| `WAIT_ACCEPT` | 待接单 | 用户已提交订单，等待老板确认 |
| `WAIT_PICKUP` | 待自提 | 老板已确认，等待用户到店自取 |
| `WAIT_DELIVERY` | 待配送 | 老板已确认，等待配送到用户地址 |
| `DELIVERING` | 配送中 | 商品已在配送途中 |
| `COMPLETED` | 已完成 | 订单已取货/送达，流程结束 |
| `CANCELED` | 已取消 | 订单已取消，流程结束 |
| `REFUNDING` | 退款中 | V1 暂不启用，保留定义 |
| `REFUNDED` | 已退款 | V1 暂不启用，保留定义 |

### 1.3 V1 简化后的状态转换规则

```typescript
const ORDER_STATE_MACHINE: Record<OrderStatus, AllowedTransitions> = {
  // 接单后直接进入待自提/待配送（跳过 PAID_WAIT_MAKE、MAKING）
  WAIT_ACCEPT: {
    WAIT_PICKUP: 'confirm',
    WAIT_DELIVERY: 'confirm',
    CANCELED: 'cancel'
  },
  WAIT_PICKUP: {
    COMPLETED: 'pickup_complete'
  },
  WAIT_DELIVERY: {
    DELIVERING: 'start_delivery'
  },
  DELIVERING: {
    COMPLETED: 'delivery_complete'
  },
  COMPLETED: {},
  CANCELED: {},
  REFUNDED: {},
  // REFUNDING 保留（V1 暂不启用但代码保留）
  REFUNDING: {
    REFUNDED: 'refund_success',
    PAID_WAIT_MAKE: 'reject_or_fail_refund'
  }
};
```

### 1.4 V1 简化说明

根据商家最新要求，V1 阶段简化订单流转：

1. **移除中间状态**：`PAID_WAIT_MAKE`（已支付/待制作）、`MAKING`（制作中）状态跳过
2. **接单即确认**：老板确认接单后，根据取货方式直接进入 `WAIT_PICKUP` 或 `WAIT_DELIVERY`
3. **V1 不接入微信支付**：无 `WAIT_PAY` 状态，支付状态标记为线下确认
4. **取货/配送简化**：自提订单确认取货即完成；配送订单经配送中到达已完成

## 二、各状态下的操作权限

### 2.1 用户端操作

| 状态 | 可操作 | 说明 |
|------|--------|------|
| WAIT_ACCEPT | 取消订单 | 待接单状态用户可取消 |
| WAIT_PICKUP | 查看订单 | 等待自提，无操作 |
| WAIT_DELIVERY | 查看订单 | 等待配送，无操作 |
| DELIVERING | 查看订单 | 配送中，无操作 |
| COMPLETED | 售后反馈（3天内） | 完成后 3 天内可提交售后问题 |
| CANCELED | 查看订单 | 已取消，仅查看 |

### 2.2 后台端操作

| 状态 | 可操作 | 说明 |
|------|--------|------|
| WAIT_ACCEPT | 确认接单 / 取消订单 | 确认时根据 pickupType 决定目标状态 |
| WAIT_PICKUP | 确认取货 | 用户到店自取，订单完成 |
| WAIT_DELIVERY | 开始配送 | 开始配送流程 |
| DELIVERING | 确认送达 | 配送完成 |
| COMPLETED | 处理售后 | 查看/回复用户的售后反馈 |

## 三、状态机校验实现

### 3.1 核心校验函数

```typescript
// backend/src/utils/stateMachine.ts

export function assertStateTransition(
  currentStatus: OrderStatus,
  targetStatus: OrderStatus
): void {
  const allowed = ORDER_STATE_MACHINE[currentStatus];
  if (!allowed || !(targetStatus in allowed)) {
    throw new BusinessError(
      409,
      `订单状态不允许从 ${currentStatus} 变更为 ${targetStatus}`
    );
  }
}

export function getAllowedActions(status: OrderStatus): string[] {
  return Object.values(ORDER_STATE_MACHINE[status] || {});
}
```

### 3.2 使用方式

所有涉及订单状态变更的 service 方法，必须先调用 `assertStateTransition()` 校验：

```typescript
// backend/src/services/order.ts
async function confirm(orderId: number, pickupType: string) {
  const order = await getOrder(orderId);
  const targetStatus = pickupType === 'DELIVERY' ? 'WAIT_DELIVERY' : 'WAIT_PICKUP';
  assertStateTransition(order.status, targetStatus);
  // ... 执行状态变更
  await logOperation(orderId, 'confirm', order.status, targetStatus);
}
```

## 四、订单取消规则

| 状态 | 用户能否取消 | 老板能否取消 | 说明 |
|------|-------------|-------------|------|
| WAIT_ACCEPT | 能 | 能 | 接单前双方均可取消 |
| WAIT_PICKUP | 不能 | 不能 | 已确认后不可取消 |
| WAIT_DELIVERY | 不能 | 不能 | 已确认后不可取消 |
| DELIVERING | 不能 | 不能 | 配送中不可取消 |
| COMPLETED | 不能 | 不能 | 已完成不可取消 |

> V1 阶段，用户端和后台端的取消能力限制在 `WAIT_ACCEPT` 状态。

## 五、状态历史追溯

每次状态变更均写入 `order_operation_log` 表：

| 字段 | 说明 |
|------|------|
| order_id | 关联订单 |
| operator_type | USER / ADMIN / SYSTEM |
| operator_id | 操作者 ID |
| action | 操作类型（confirm、cancel、pickup_complete 等） |
| from_status | 变更前状态 |
| to_status | 变更后状态 |
| remark | 备注（如取消原因） |

## 六、V1 保留但暂不启用的状态

以下状态及相关逻辑保留代码，但 V1 不启用：

| 状态 | 用途 | 恢复条件 |
|------|------|----------|
| `WAIT_PAY` | 待支付 | 接入微信支付后 |
| `PAID_WAIT_MAKE` | 已支付/待制作 | 需要区分支付和制作阶段时 |
| `MAKING` | 制作中 | 需要制作进度追踪时 |
| `REFUNDING` | 退款中 | 接入微信支付和线上退款后 |
| `REFUNDED` | 已退款 | 同上 |
