# 《蛋糕定制》微信小程序 PRD 产品需求文档

## 1. 产品概述

### 1.1 产品名称

《蛋糕定制》微信小程序

### 1.2 使用对象

- C 端用户：需要预约生日蛋糕、儿童蛋糕、定制蛋糕的顾客
- B 端商家：单店老板，负责接单、报价、制作、配送、售后

### 1.3 产品定位

面向小型蛋糕门店的预约型微信小程序。核心不是复杂商城，而是帮助用户快速提交蛋糕预约和定制需求，帮助老板在线确认订单、报价、线下收款、制作和配送。

### 1.4 核心目标

1. 用户可在线查看蛋糕商品和蛋糕案例。
2. 用户可提交蛋糕预约/定制需求。
3. 老板确认订单和价格后，系统直接视为已支付/待制作，实际收款由商家线下处理。
4. 老板可管理订单状态、配送、自提和售后。
5. 支持基础统计报表，方便老板查看今日订单、销售额和退款金额。

### 4.1 订单状态列表

| 状态 | 说明 |
|---|---|
| 待接单 | 用户提交需求，等待老板确认 |
| 待支付 | V1 暂不使用；后续接入微信支付后恢复 |
| 制作中 | 用户已微信支付，老板开始制作 |
| 待自提 | 蛋糕已制作完成，等待用户到店取货 |
| 待配送 | 蛋糕已制作完成，等待老板配送 |
| 配送中 | 老板正在配送 |

| 退款中 | 用户申请退款，等待老板审核 |
| 已退款 | 老板同意退款并完成退款 |

### 4.2 状态流转

> ⚠️ 以下为初版流转图，最终修正版见 [13. 最终修正规则](#13-最终修正规则)。核心差异：支付后先进入「已支付/待制作」，由老板点击"开始制作"后才进入「制作中」。

```text
待接单
  ├─ 老板确认报价/接单 → 已支付/待制作
  ├─ 老板无法接单 → 已取消
  └─ 用户取消 → 已取消

待支付
  ├─ V1 跳过线上支付，商家接单即视为已支付/待制作
  └─ 用户取消 → 已取消

已支付/待制作
  ├─ 老板点击开始制作 → 制作中
  └─ V1 不启用线上退款；线下协商处理

制作中
  ├─ 自提订单制作完成 → 待自提
  └─ 配送订单制作完成 → 待配送

待自提
  └─ 老板确认用户已取货 → 已完成

待配送
  └─ 老板开始配送 → 配送中

配送中
  └─ 老板确认送达 → 已完成

制作中之前
  └─ V1 不启用线上退款；线下协商处理 → 已退款/恢复原状态

已完成
  └─ 3 天内可申请售后
```

### 4.3 支付流程

> ⚠️ 已废弃，最终修正版见 [13. 最终修正规则](#13-最终修正规则)。支付成功后订单进入「已支付/待制作」，需老板点击"开始制作"后才进入「制作中」。

```text
用户提交需求
→ 待接单
→ 老板确认报价
→ 待支付
→ V1 跳过线上支付（后续接入微信支付时恢复）
→ 支付成功
→ 制作中
```

V1 不记录微信支付交易号；可记录线下确认时间和线下确认支付状态，用于订单流转和报表统计。

## 5. 退款与售后规则

### 5.1 退款原则

1. 所有蛋糕均为现做。
2. 未付款订单可直接取消。
3. V1 不提供线上退款入口；线下收款后的退款、改期由商家与用户自行协商。
4. 一旦进入“制作中”，不支持退款。
5. 用户填写错误、临时改期、联系不上导致无法交付，不支持退款。
6. 配送延误不支持退款。
7. 售后入口保留 3 天。
8. 售后只做问题反馈，不做自动退款。

### 5.2 退款规则表

| 场景 | 是否支持 | 处理方式 |
|---|---|---|
| 待接单未付款 | 支持取消 | 用户直接取消 |
| 待支付未付款 | 支持取消 | 用户直接取消 |
| 已支付/待制作 | V1 不展示线上退款入口 | 线下协商处理 |
| 已进入制作中 | 不支持退款 | 页面提示不可退款 |
| 用户填错文字 | 原则不退款 | 老板可联系用户确认 |
| 用户临时改期 | 原则不退款 | 可线下协商 |
| 联系不上用户 | 不退款 | 责任归用户 |
| 配送延误 | 不退款 | 当前规则不支持退款 |
| 已完成 | 不支持退款，仅支持售后反馈 | 3 天内可申请售后 |

### 5.3 售后规则

| 项目 | 规则 |
|---|---|
| 售后入口保留 | 订单完成后 3 天 |
| 售后类型 | 用户反馈问题 |
| 处理人 | 老板 |
| 是否自动退款 | 否 |
| 处理方式 | 老板查看反馈后线下处理 |

## 6. 配送规则

### 6.1 配送范围

仅支持门店周边 5km 内配送。

### 6.2 配送费

配送费按距离计算：

```text
配送费 = 四舍五入后的公里数 × 2 元
```

建议不足 1km 按 1km 计算。

> ⚠️ 已废弃，最终修正版见 [13.3](#133-配送费规则)。配送费 = `ROUND(实际距离) × 2元`。四舍五入后为 0km 时配送费为 0 元。

| 实际距离 | 计费公里 | 配送费 |
|---:|---:|---:|
| 0.4km | 0km | 0元 |
| 0.6km | 1km | 2元 |
| 1.4km | 1km | 2元 |
| 1.5km | 2km | 4元 |
| 4.5km | 5km | 10元 |
| 超过5km | - | 不支持配送 |

### 6.3 配送地址

1. 用户选择商家配送时，必须填写配送地址。
2. 小程序使用微信内置定位能力，根据门店定位和用户地址自动计算距离。
3. 超出 5km 时，提示用户选择到店自提。

### 6.4 老板修改配送费

老板确认订单时：

| 字段 | 是否可修改 |
|---|---|
| 尺寸/磅数 | 不可修改 |
| 口味 | 不可修改 |
| 配送地址 | 不可修改 |
| 配送费 | 可修改 |
| 订单总价 | 可修改 |
| 老板备注 | 可填写 |

## 13. 最终修正规则

以下规则为最终口径，覆盖本文档前文中与之不一致的内容。

### 13.1 订单状态

订单状态最终调整为：

```text
待接单、待支付、已支付/待制作、制作中、待自提、待配送、配送中、已完成、已取消、退款中、已退款
```

核心流转：

```text
用户提交需求 → 待接单
老板确认报价/接单 → 已支付/待制作
V1 跳过线上支付，商家接单即视为已支付/待制作
老板点击开始制作 → 制作中
制作完成：
  自提订单 → 待自提 → 已完成
  配送订单 → 待配送 → 配送中 → 已完成
```

### 13.2 退款规则

V1 不展示线上退款入口；后续接入微信支付后，才恢复“已支付/待制作可申请退款，制作中后不可退款”。

退款流程：

```text
V1 不启用线上退款；线下协商处理
老板同意退款 → V1 不调用微信原路退款，线下协商处理
微信退款成功回调 → 已退款
```

老板拒绝退款后，订单恢复为 `已支付/待制作`。

### 13.3 配送费规则

配送费按实际距离四舍五入后的公里数计费：

```text
配送费 = 四舍五入后的公里数 × 2元
```

配送范围判断仍按实际距离：

```text
实际距离 <= 5km：支持配送
实际距离 > 5km：不支持配送
```

### 13.4 非营业时间预约

用户允许预约非营业时间取货/送达。用户选择 06:00-18:00 之外的时间时，前端提示：

```text
当前预约时间为非营业时间，请确认商家可安排后提交。
```

后台订单列表和订单详情增加 `非营业时间预约` 标签。

### 13.5 联系规则

老板微信客服首版暂不接入。用户端客服入口提示 `客服功能暂未接入`。老板通过订单中的用户手机号线下联系用户，用于确认祝福文字、备注、地址等信息。

### 13.6 图片资源规则

当前没有真实蛋糕图片时，V1.0 采用以下方案：

1. 商品和案例后台仍保留图片必填规则。
2. 开发和 UI 阶段使用统一蛋糕占位图，避免页面空白。
3. 上线前由商家补充真实蛋糕图片后再发布。
4. 商品图片建议使用门店实际成品图，不建议长期使用网图。
5. 蛋糕案例图库必须使用商家真实案例图或获得授权的图片。

图片规格建议：

| 类型 | 建议尺寸 | 数量 |
|---|---|---|
| 商品主图 | 800×800 | 每个商品 1 张 |
| 案例图片 | 800×800 或 1080×1080 | 每个案例 1 张 |
| 用户参考图 | 不限制比例 | 最多 6 张 |
| 售后图片 | 不限制比例 | 最多 6 张 |

## V1 临时支付链路调整（最终口径）

因商家暂不注册个体工商户、暂不接入微信支付商户号，V1 当前支付链路调整如下。本节为最终口径，覆盖本文档中所有“待支付、微信支付、支付回调、微信原路退款”的旧描述。

### 1. 当前订单主流程

```text
用户提交订单 → 待接单
商家确认报价/接单 → 已支付/待制作
商家点击开始制作 → 制作中
制作完成：
  自提订单 → 待自提 → 已完成
  配送订单 → 待配送 → 配送中 → 已完成
```

### 2. 支付处理规则

1. V1 不接入微信支付，不调用 `uni.requestPayment`，不创建微信预支付单。
2. 商家在后台点击确认报价/接单后，系统直接将订单状态置为 `PAID_WAIT_MAKE`（页面文案：已支付/待制作）。
3. 该“已支付”是业务上的线下确认状态，不代表微信支付真实到账。
4. 后端可写入 `pay_status = PAID`、`payment_method = OFFLINE`、`paid_at = accepted_at`，用于驱动现有订单状态和报表逻辑。
5. 如需保留支付记录，可创建一条本地记录，状态建议为 `OFFLINE_CONFIRMED`；不得生成微信 `transaction_id`。
6. 报表中的销售额按“商家确认后的订单金额”统计，不等同于微信支付流水或真实到账金额。

### 3. 退款处理规则

1. V1 用户端不展示申请退款入口。
2. V1 后台不展示退款审核入口，退款列表可隐藏或置灰。
3. 退款接口和微信原路退款代码暂不启用；如保留接口，应返回“当前暂不支持线上退款”。
4. 线下退款、改期、取消等由商家与用户自行协商，不进入小程序自动退款流程。
5. 售后反馈仍可保留，但只作为问题反馈，不自动退款。

### 4. 后续恢复微信支付时的切换点

后续商家完成主体认证并开通微信支付后，再恢复以下链路：

```text
商家确认报价/接单 → 待支付
V1 跳过线上支付成功回调（V1 暂不启用） → 已支付/待制作
```

届时再启用微信预支付、支付回调、微信原路退款和真实支付流水统计。
